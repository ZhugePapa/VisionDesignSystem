这篇文章 **《The full stack design system》** 是设计系统领域非常经典的一篇文章，作者是 Emmet Connolly，发表于 Intercom 官方博客。它提出了一个非常重要的观点：

**设计系统不应该只是 UI 组件库，而应该成为贯穿产品概念、设计、代码、团队语言的一整套产品系统。**  

结合你现在正在做的 **军工 DevOps Design System、UX 知识库、组件体系建设**，这篇文章其实非常值得深入理解，因为它解决的是很多企业级设计系统后期都会遇到的问题：

“组件越来越多，但是产品越来越乱。”

---

# **一、文章核心观点一句话总结**

传统 Design System：

「让按钮、表单、卡片长得一样」

Full Stack Design System：

「让整个产品用同一种方式思考」

它认为：

**UI 一致性 ≠ 产品一致性**

很多设计系统解决了第一层，但是没有解决第二层。

---

# **二、文章批判的问题：Pattern Library 不等于 Design System**

文章首先提出一个问题：

很多公司所谓的 Design System，其实只是：

Pattern Library（模式库）

比如：

```
Button
Input
Select
Table
Modal
Card
```

然后设计师和开发：

拖出来组合。

类似：

```
     Button
       |
Input--+--Card
       |
     Table
```

问题：

这些组件本身没有业务认知。

例如：

Button：

```
type:
- primary
- danger
- secondary
```

但是系统不知道：

这个按钮用于：

- 创建需求？
- 发布版本？
- 删除制品？
- 审批流程？
- 部署环境？

所以最后：

组件统一了：

✅ 颜色统一  
✅ 圆角统一  
✅ spacing统一

但是产品体验：

❌ 不统一

文章用了一个 Lego 比喻：

普通组件库像一桶乐高积木：

你可以搭任何东西。

但是也可能搭出：

一个看起来完全不像一个产品的东西。

而真正的设计系统更像：

一个指定模型的 Lego 套件。

组件知道自己应该组成什么。

---

# **三、Full Stack Design System 的四层结构**

这是全文最重要的部分。

它认为完整设计系统应该覆盖四层：

---

# **Layer 1：Product Concept（产品概念层）**

这是最容易被忽略的一层。

不是：

```
Button
Input
Table
```

而是：

```
用户
项目
任务
需求
版本
发布
审批
环境
制品
```

也就是：

产品里的核心对象 Object。

例如：

Intercom：

核心对象：

```
Customer
Conversation
Message
Article
Teammate
```

Facebook：

```
Friend
Post
Message
Group
Event
```

Slack：

```
Team
Channel
Message
Thread
Reaction
```

这些不是 UI。

这些是：

**产品世界里的名词。**

---

## **对你的军工 DevOps 来说：**

传统组件思考：

```
Table
Form
Tree
Dialog
```

Full Stack 思考：

```
Requirement
Issue
Build
Pipeline
Artifact
Approval
Environment
Security Gate
Release
```

例如：

一个：

「审批组件」

不要定义：

```
ApprovalCard
ApprovalButton
ApprovalDialog
```

而应该定义：

```
Approval Object
```

然后：

它在不同层出现：

### **产品层**

```
需求审批
 ↓
技术评审
 ↓
安全审查
 ↓
发布批准
```

---

### **设计层**

```
Approval View

- Pending
- Approved
- Rejected
- Expired
```

---

### **组件层**

```
<ApprovalStatus />

<ApprovalTimeline />

<ApprovalAction />

<ApprovalDetail />
```

---

### **代码层**

```vue
<ApprovalObject
 status="pending"
/>
```

这就是 Full Stack。

---

# **Layer 2：Shared Language（统一语言）**

文章非常强调：

Words matter.

也就是：

团队必须共享产品语言。

比如：

设计师说：

发布任务

开发说：

Deploy Job

测试说：

Delivery Pipeline

用户说：

发布版本

如果没有统一：

每个人脑中都是不同模型。

---

Full Stack Design System 会建立：

```
Object Dictionary
```

例如：

## **Artifact**

定义：

软件构建过程中产生的可部署文件集合。

包含：

属性：

```
name
version
repository
checksum
securityLevel
```

状态：

```
created
scanning
verified
released
deprecated
```

相关对象：

```
Artifact
 |
 + Build
 |
 + Pipeline
 |
 + Release
```

---

这其实非常接近：

DDD（领域驱动设计）

中的：

Ubiquitous Language（统一语言）

---

# **Layer 3：Design Resource（设计资产层）**

这里对应 Figma。

传统：

```
Button.fig
Input.fig
Table.fig
```

Full Stack：

应该是：

```
Artifact.fig
Pipeline.fig
Approval.fig
```

例如：

Figma 中：

不是画：

```
一个表格
```

而是：

```
Artifact List View
```

里面包含：

- artifact 状态
- 安全等级
- 扫描结果
- 操作权限
- 发布状态

---

# **Layer 4：Code Component（代码组件层）**

设计和代码一一对应。

文章强调：

Sketch 中的 Object 应该对应代码中的 Object。

例如：

设计：

```
Conversation
```

代码：

```tsx
<Conversation />
```

而不是：

设计：

```
聊天列表
聊天卡片
聊天详情
```

代码：

```
ChatList
ChatCard
ChatDetail
```

这样会导致：

一个业务对象：

被拆散。

# **四、它和 Atomic Design 的区别**

这是文章最关键的理论贡献。

---

## **Atomic Design**

Atomic Design：

结构：

```
Atom

 ↓

Molecule

 ↓

Organism

 ↓

Template

 ↓

Page
```

例如：

```
Button

+
Input

=
Login Form
```

优点：

组件复用。

缺点：

不知道为什么存在。

---

Full Stack：

结构：

```
Product Object

↓

Experience

↓

Component

↓

Code
```

例如：

不是：

```
Button
 +
Form
 +
Table
```

而是：

```
Release

↓

Release Detail

↓

Release Components

↓

Code
```

---

# **五、为什么 Intercom 认为这个有效？**

文章总结三个价值：

---

## **1. 防止产品腐化**

大型产品的问题：

```
1年后

10个页面

20种按钮

30种命名

50种流程
```

最后：

没人知道哪个是正确的。

Full Stack：

所有东西围绕：

```
核心对象
```

增长。

---

## **2. 新功能开发更快**

传统：

需求来了：

设计新页面。

↓

设计组件。

↓

开发。

↓

出现新的模式。

Full Stack：

问：

这个需求属于哪个 Object？

例如：

新增：

“自动安全扫描”

不是创建：

```
Security Scan Page
```

而是：

扩展：

```
Pipeline Object
```

增加：

```
securityStatus
scanResult
```

---

## **3. 用户学习成本降低**

文章说：

用户不应该学习：

100个功能。

而应该理解：

10个核心对象。

例如：

Slack：

用户理解：

```
Channel
Message
Thread
```

然后所有功能围绕它们展开。

# **总结**

这篇文章真正改变的是：

以前：

Design System = UI 组件复用系统

Intercom 提出：

Design System = 产品认知系统

对于你正在建设的军工 DevOps Design System，我认为应该采用：

**Full Stack Design System + DDD + Object View**

而不是单纯：

**Element Plus 二次封装组件库。**

你的设计系统核心资产未来应该不是：

```
Button
Table
Form
```

而应该是：

```
Pipeline
Artifact
Requirement
Approval
Security Gate
Release
```

这些业务对象。

这也是为什么 Linear、GitHub、Atlassian 这类复杂产品的设计系统，最终都会从“组件库”走向“产品系统”。 

我觉得下一步非常适合继续拆解一个问题：

**“Full Stack Design System 如何落地到军工 DevOps：从 Object Model → Figma → Vue Component → Storybook 的完整架构。”** 这会和你现在做的设计系统非常契合。
