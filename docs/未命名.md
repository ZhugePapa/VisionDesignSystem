可以把 **Activity Feed** 定义成：面向军工 DevOps 场景的「过程留痕 + 协作讨论 + 审批审计」业务组件。它不只是普通时间线，而是承载研发活动、审批动作、闸门流转、质量检查、制品变更、审计证据的统一活动流。

## 一、Activity Feed 应具备的核心功能

### 1. 活动记录展示

用于展示 DevOps 对象上的所有关键行为，例如：

|活动类型|示例|
|---|---|
|创建类|创建需求、创建缺陷、创建任务、创建合并请求|
|状态变更|从“开发中”变为“待评审”|
|分派变更|指派给某人、移除负责人|
|评论讨论|普通评论、回复、@某人|
|代码活动|提交代码、关联 commit、创建分支、合并代码|
|CI/CD 活动|构建开始、构建成功、构建失败、部署完成|
|质量活动|单测通过率、代码扫描、安全扫描、质量门禁|
|审批活动|提交审批、通过、驳回、撤回、加签、转交|
|闸门活动|进入 Gate、Gate 通过、Gate 阻塞、补充材料|
|制品活动|生成制品、发布制品、归档制品、下载制品|
|基线活动|纳入基线、基线变更、版本冻结|
|审计活动|查看记录、导出记录、权限变更、涉密级别变更|

---

### 2. 军工行业特有能力

军工 DevOps 的 Activity Feed 不能只关注协作效率，还要关注 **合规、留痕、可追溯、权限隔离**。

建议重点支持这些能力：

|能力|说明|
|---|---|
|全流程留痕|所有关键动作都要有记录，不允许随意删除|
|审批链展示|清晰展示谁提交、谁审批、谁驳回、理由是什么|
|证据材料关联|每个活动可关联检查报告、测试报告、评审记录、制品等|
|涉密级别标识|活动、附件、评论可带密级标签|
|操作人身份可信|展示真实姓名、组织、角色、工号、签名状态|
|不可篡改标记|可展示 hash、签名、归档状态、审计状态|
|权限可见性控制|不同角色看到的活动范围不同|
|关键节点高亮|Gate、审批、基线、发布、驳回等节点需要突出|
|审计导出|支持导出为审计报告、PDF、Excel 或归档记录|
|离线/内网适配|适合私有化、内网、无公网环境|

---

### 3. 信息过滤与检索

Activity Feed 通常会非常长，所以需要提供筛选能力。

建议支持：

|功能|说明|
|---|---|
|按活动类型筛选|只看评论、审批、构建、闸门、制品等|
|按人员筛选|查看某个人的操作记录|
|按时间筛选|今天、本周、指定时间范围|
|按重要程度筛选|普通、重要、阻塞、风险|
|按审计属性筛选|是否可审计、是否已归档、是否包含证据|
|按密级筛选|公开、内部、秘密、机密等|
|搜索关键词|搜索评论、审批意见、制品名称、提交信息|

---

### 4. 评论与协作

如果 Activity Feed 同时承担讨论区能力，需要支持：

|功能|说明|
|---|---|
|发表评论|普通文本、富文本、Markdown|
|回复评论|支持 threaded reply|
|@人员|提醒相关人|
|附件上传|上传评审材料、截图、报告|
|评论引用|引用某条活动进行讨论|
|编辑评论|受权限控制|
|删除评论|军工场景建议软删除，并保留审计记录|
|内部备注|仅特定角色可见|
|处理意见|审批/驳回时填写意见|

---

### 5. 状态与风险表达

Activity Feed 需要把不同活动的状态表达清楚。

建议支持这些状态：

|状态|示例|
|---|---|
|success|构建成功、审批通过|
|warning|存在风险、缺少材料|
|error|构建失败、扫描不通过|
|pending|等待审批、等待检查|
|blocked|闸门阻塞、流程卡住|
|info|普通信息|
|archived|已归档、已纳入基线|
|classified|涉密活动|

## 二、建议的组件结构

Activity Feed 不建议做成一个巨大的单组件，可以拆成以下结构：

```ts
ActivityFeed
├── ActivityFeedHeader
├── ActivityFeedFilter
├── ActivityFeedList
│   └── ActivityItem
│       ├── ActivityIcon
│       ├── ActivityHeader
│       ├── ActivityBody
│       ├── ActivityMeta
│       ├── ActivityEvidence
│       └── ActivityActions
├── ActivityComposer
└── ActivityAuditFooter
```

### 推荐组件分层

|组件|作用|
|---|---|
|ActivityFeed|整体业务组件|
|ActivityItem|单条活动记录|
|ActivityComposer|评论/处理意见输入区|
|ActivityFilter|筛选与搜索|
|ActivityEvidence|证据材料展示|
|ActivityAuditMeta|审计信息展示|
|ApprovalActivityItem|审批活动专用展示|
|GateActivityItem|闸门活动专用展示|
|BuildActivityItem|构建活动专用展示|
|ArtifactActivityItem|制品活动专用展示|

## 三、Activity 数据结构设计

核心不是先设计 props，而是先设计 **ActivityItem 数据模型**。

```ts
type ActivityType =
  | 'comment'
  | 'reply'
  | 'create'
  | 'update'
  | 'status_change'
  | 'assign'
  | 'commit'
  | 'merge_request'
  | 'build'
  | 'deploy'
  | 'test'
  | 'scan'
  | 'approval'
  | 'gate'
  | 'artifact'
  | 'baseline'
  | 'release'
  | 'permission'
  | 'audit'
  | 'system';

type ActivityStatus =
  | 'info'
  | 'success'
  | 'warning'
  | 'error'
  | 'pending'
  | 'blocked'
  | 'archived'
  | 'classified';

type ClassificationLevel =
  | 'public'
  | 'internal'
  | 'secret'
  | 'confidential'
  | 'top_secret';

interface ActivityActor {
  id: string;
  name: string;
  avatar?: string;
  department?: string;
  role?: string;
  employeeNo?: string;
  organizationPath?: string;
}

interface ActivityEvidence {
  id: string;
  name: string;
  type: 'file' | 'report' | 'artifact' | 'commit' | 'pipeline' | 'test_case' | 'document';
  url?: string;
  status?: 'valid' | 'invalid' | 'expired' | 'archived';
  classification?: ClassificationLevel;
}

interface ActivityItemData {
  id: string;
  type: ActivityType;
  status?: ActivityStatus;

  actor: ActivityActor;
  action: string;
  target?: {
    id: string;
    name: string;
    type: string;
    url?: string;
  };

  title?: string;
  content?: string;
  richContent?: string;

  createdAt: string;
  updatedAt?: string;

  before?: unknown;
  after?: unknown;

  evidence?: ActivityEvidence[];

  approval?: {
    approvalId: string;
    stepName: string;
    result: 'approved' | 'rejected' | 'pending' | 'withdrawn' | 'transferred' | 'countersigned';
    opinion?: string;
    approvers?: ActivityActor[];
  };

  gate?: {
    gateId: string;
    gateName: string;
    result: 'passed' | 'failed' | 'blocked' | 'pending';
    checklist?: {
      id: string;
      name: string;
      status: 'passed' | 'failed' | 'pending';
    }[];
  };

  audit?: {
    traceId?: string;
    signatureStatus?: 'signed' | 'unsigned' | 'invalid';
    hash?: string;
    archived?: boolean;
    immutable?: boolean;
    operatorIp?: string;
    deviceId?: string;
  };

  classification?: ClassificationLevel;

  permissions?: {
    canView?: boolean;
    canReply?: boolean;
    canEdit?: boolean;
    canDelete?: boolean;
    canExport?: boolean;
    canViewAuditDetail?: boolean;
  };

  children?: ActivityItemData[];
}
```

```ts
   */

  /**

   */
  groupByDate?: boolean;

  /**
   * 是否展示筛选器

  showFilter?: boolean;

  /**
   * 是否展示搜索框
   */
  searchable?: boolean;

  /**
   * 是否展示评论输入框
   */
  showComposer?: boolean;

  /**
   * 是否允许回复
   */
  replyable?: boolean;

  /**
   * 是否允许编辑评论
   */
  editable?: boolean;

  /**
   * 是否允许删除评论
   */
  deletable?: boolean;

  /**
   * 是否允许导出审计记录
   */
  exportable?: boolean;

  /**
   * 是否展示审计信息
   */
  showAuditMeta?: boolean;

  /**
   * 是否展示密级标签
   */

   * 是否展示证据材料
   */

   * 是否展示操作按钮
   */

   * 当前用户
   */

    id: string;
    type: string;
    name: string;
```

## 九、设计上的关键建议

Activity Feed 在军工 DevOps 里建议分成三种视觉层级：

|层级|用途|视觉建议|
|---|---|---|
|普通活动|评论、状态变更、指派|轻量展示|
|重要活动|审批、驳回、构建失败、扫描失败|高亮状态、突出原因|
|审计活动|密级变更、权限变更、基线归档、发布记录|稳重、不可编辑、显示审计标识|

最重要的是：**Activity Feed 不要只做“发生了什么”，而要表达“谁在什么时间，基于什么权限，对什么对象，做了什么动作，产生了什么结果，是否形成可审计证据”。**

这句话可以作为这个业务组件的设计原则。
